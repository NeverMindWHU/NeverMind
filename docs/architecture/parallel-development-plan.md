# NeverMind 前后端并行开发技术方案

## 1. 目标

本方案用于支持 NeverMind 在 `Tauri v2 + Rust + 前端 Web` 架构下实现前后端并行开发，降低相互阻塞，提升迭代效率，并保证联调阶段的稳定性与可控性。

核心目标如下：

- 前端在 Rust 后端未完成时，依然可以独立推进页面、交互和状态管理开发。
- Rust 后端在前端页面未完成时，依然可以独立推进命令接口、调度逻辑、数据库和通知能力开发。
- 双方通过稳定的接口契约协作，而不是通过口头约定协作。
- 在功能演进过程中，避免因接口变更、数据结构漂移和数据库升级导致反复返工。

---

## 2. 总体思路

NeverMind 采用以下并行开发策略：

1. 契约先行：先定义前后端共享的数据结构、命令名、输入输出格式和错误模型。
2. 前端解耦：前端通过 `Service Adapter` 调用能力层，不直接耦合真实 Tauri API。
3. Mock 驱动：在 Rust 能力未完成前，由前端使用 Mock Service 和固定假数据推进开发。
4. 后端独立：Rust 后端按 Command、Service、DAO、Scheduler 分层开发，可独立测试。
5. 分阶段联调：先联调静态返回，再联调真实数据库，再联调 AI 和通知等异步能力。

该策略的关键不是“谁等谁”，而是让前端和后端都围绕同一份契约同时推进。

---

## 3. 推荐分层

### 3.1 前端侧

前端建议继续使用当前目录思路：

- `src/features/*`：按业务模块组织页面、组件、hooks、services。
- `src/lib/`：封装基础调用能力，例如调用适配器、错误处理、时间格式化等。
- `src/store/`：存放全局状态管理，如当前复习任务、用户设置、主题状态等。
- `src/types/`：存放前端领域模型、DTO 类型和 UI 状态类型。

前端不应直接在组件中调用 Tauri 原生接口，而应统一通过 `services` 层访问。

### 3.2 后端侧

Rust 侧建议按职责边界实现：

- `src-tauri/src/commands/`：暴露给前端的 Tauri Commands。
- `src-tauri/src/ai/`：AI 请求封装、模型适配、重试与超时策略。
- `src-tauri/src/scheduler/`：艾宾浩斯复习算法与排期规则。
- `src-tauri/src/db/dao/`：数据库访问层。
- `src-tauri/src/models/`：核心实体和传输结构。
- `src-tauri/src/notifications/`：桌面通知能力。
- `src-tauri/src/tray/`：系统托盘菜单和行为管理。
- `src-tauri/src/state/`：应用运行时共享状态。

`commands` 只负责参数接收、结果返回和错误映射，不直接承担复杂业务逻辑。

---

## 4. 并行开发的核心机制

### 4.1 契约先行

每个可独立交付的功能模块，都需要先定义一份“接口契约”，至少包含以下内容：

- Command 名称
- 输入参数
- 输出结构
- 错误码或错误类型
- 字段含义
- 是否涉及异步任务
- 是否涉及本地数据库写入

建议在 `docs/architecture/` 下维护契约文档，也可以后续补充一份共享契约目录，例如：

```text
docs/architecture/contracts/
├── card-generation.md
├── review.md
├── library.md
└── settings.md
```

一个功能契约建议包含如下内容：

```text
Command: generate_card
Input:
- source_text: string
- selected_keyword: string

Output:
- card_id: string
- keyword: string
- definition: string
- explanation: string
- mnemonic: string
- next_review_at: string

Errors:
- AI_TIMEOUT
- AI_UNAVAILABLE
- INVALID_INPUT
- DB_WRITE_FAILED
```

有了契约以后：

- 前端根据输入输出定义页面状态、表单、列表、错误提示。
- 后端根据同一份定义实现命令、模型、数据库写入与测试。

### 4.2 Adapter 适配层

前端不要在组件里直接调用 `@tauri-apps/api` 或 `invoke()`，而是通过统一适配层完成。

建议调用链如下：

```text
UI Component
   -> Feature Hook
   -> Service
   -> Adapter
   -> Tauri Command / Mock Service
```

这样可以支持两套运行模式：

1. `mock` 模式：前端直接返回本地假数据。
2. `tauri` 模式：前端通过 Tauri IPC 调用 Rust 后端。

这样前端的页面、交互、错误态、加载态、空态都能在后端未完成前先做完。

### 4.3 Mock 数据驱动

对于以下模块，建议前端优先基于 Mock 数据开发：

- 知识卡片生成结果展示
- 复习卡片流转
- 知识宝库筛选与搜索
- 设置页面

Mock 数据建议保持“结构真实、内容简化”的原则：

- 字段名必须与正式契约一致。
- 时间字段格式与真实返回一致。
- 错误返回结构也要模拟。
- 不要为了快而定义与正式接口不一致的临时字段。

建议未来补充类似目录：

```text
src/
├── mocks/
│   ├── cards.ts
│   ├── reviews.ts
│   └── settings.ts
```

### 4.4 后端独立可测

Rust 后端开发不依赖前端页面完成，建议按以下顺序推进：

1. 先写领域模型和 DTO。
2. 再写 DAO 和迁移脚本。
3. 再写 Service 逻辑。
4. 最后暴露为 Tauri Commands。

测试顺序建议如下：

1. 单元测试：复习排期算法、输入校验、错误映射。
2. 集成测试：数据库读写、迁移执行、命令调用。
3. 联调验证：通过前端实际页面验证交互流程。

这样后端可以在没有前端页面的情况下，先把核心逻辑跑通。

## 5. 模块级并行方案

### 5.1 卡片生成模块

前端负责：

- 关键词输入或文本选择后的交互流程
- 生成中状态、生成失败提示、卡片预览与确认保存
- 卡片字段编辑体验

后端负责：

- `generate_card` 命令
- AI 请求调度
- 生成结果标准化
- 入库逻辑
- 默认复习计划初始化

并行方式：

- 前端先用固定假数据模拟 AI 返回。
- 后端先完成纯命令返回，再逐步接入真实模型。

### 5.2 复习模块

前端负责：

- 正反面翻卡交互
- `记住了 / 未记住 / 跳过` 按钮流程
- 今日待复习列表和完成反馈

后端负责：

- 获取待复习卡片列表
- 根据反馈更新下次复习时间
- 复习记录写入
- 定时任务与通知触发

并行方式：

- 前端先根据固定复习队列开发翻卡体验。
- 后端先实现排期算法和反馈计算逻辑。

### 5.3 知识宝库模块

前端负责：

- 列表页、搜索框、筛选器、详情抽屉或详情页
- 批量选择、导出入口、排序和分页体验

后端负责：

- 卡片列表查询
- 关键词检索
- 分类标签过滤
- 导入导出接口

并行方式：

- 前端先基于分页 Mock 数据完成列表交互。
- 后端先确定筛选参数和返回结构。

### 5.4 设置模块

前端负责：

- 模型配置表单
- 通知偏好设置
- 主题和基础偏好界面

后端负责：

- 设置读取与持久化
- 模型配置校验
- 系统通知能力探测

并行方式：

- 前端先用本地假配置完成表单和保存反馈。
- 后端补齐真实设置存储和读取逻辑。

## 6. 接口与数据约定

为了减少联调返工，建议统一以下技术约定：

- 命令命名：使用清晰的动词加领域名，例如 `generate_card`、`list_due_reviews`、`update_review_result`。
- 返回结构：统一使用稳定字段，不在不同模块中随意改变命名风格。
- 时间格式：统一使用 ISO 8601 字符串。
- 主键格式：统一使用 `UUID` 或等价唯一标识。
- 错误结构：统一返回 `code`、`message`、`details`。
- 可空字段：在契约中明确标记，不依赖前端猜测。

推荐统一响应风格：

```text
Success:
{
  success: true,
  data: ...
}

Error:
{
  success: false,
  error: {
    code: "...",
    message: "...",
    details: ...
  }
}
```

如果未来需要多端扩展，这种统一风格也有利于迁移到移动端或 Web 版本。

## 7. 数据库迁移协作方案

由于 NeverMind 是本地优先应用，数据库结构演进必须纳入并行开发方案。

约定如下：

1. 每次数据库结构变化，必须新增 migration 文件，不直接修改旧版本脚本。
2. 每次新增字段或表时，同时更新契约文档和数据模型说明。
3. 前端若依赖新字段，必须明确标记最低后端版本或 migration 版本。
4. 联调前，先确认本地数据库已升级到目标版本。

建议 migration 命名方式：

```text
001_init.sql
002_create_review_schedule.sql
003_add_card_tags.sql
004_add_settings_table.sql
```

这样可以保证前端、后端、数据库三者在同一版本线上演进。

## 8. 联调流程

建议把联调分为四个阶段：

### 阶段一：契约联调

- 双方只确认字段、命令、错误结构。
- 不要求真实数据库与真实 AI。
- 目标是确保双方理解一致。

### 阶段二：静态联调

- Rust 返回固定数据。
- 前端开始接入真实命令调用。
- 目标是验证参数传递、数据渲染和异常提示。

### 阶段三：真实数据联调

- 接入 SQLite。
- 验证增删改查、搜索、筛选、排期计算。
- 目标是验证核心业务闭环。

### 阶段四：异步能力联调

- 接入 AI、通知、托盘、定时任务。
- 验证超时、失败重试、权限问题和后台行为。
- 目标是验证桌面应用特有能力。

这种分阶段方式可以显著降低“一上来全量联调”带来的问题定位难度。

## 9. 分支与协作建议

建议采用功能模块分支，而不是纯前后端分支：

- `feature/card-generation`
- `feature/review-flow`
- `feature/library-management`
- `feature/settings`

每个功能分支内，再由前端和后端分别提交各自实现，但共同围绕同一份功能契约工作。

提交顺序建议：

1. 先提交契约文档。
2. 再提交前端 Mock 版本。
3. 再提交后端真实能力。
4. 最后提交联调修正。

这样更容易 review，也更容易看出一个功能是否真正闭环。

---

## 10. 推荐落地步骤

对于 NeverMind 当前阶段，建议按以下顺序落地：

1. 先补齐 4 个核心模块的契约文档：卡片生成、复习、知识宝库、设置。
2. 前端先搭页面骨架、状态流转和 Mock Service。
3. Rust 后端先搭 `commands + models + dao + scheduler` 空实现和测试框架。
4. 先完成知识宝库与设置模块联调，因为它们依赖最简单。
5. 再完成复习模块联调，因为涉及算法与状态回写。
6. 最后完成卡片生成模块联调，因为它依赖 AI 服务和错误处理策略。

---

## 11. 最终建议

对于 NeverMind 这种桌面端本地优先项目，最适合的并行开发方式不是传统 Web 项目的“前端等接口”，而是：

- 用契约文档固定边界
- 用 Adapter 隔离调用方式
- 用 Mock 保证前端不停工
- 用分层测试保证后端不停工
- 用分阶段联调降低集成风险

如果后续团队规模扩大，可以在此基础上继续补充：

- 共享 DTO 生成方案
- 自动化契约校验
- 数据库版本校验脚本
- 联调环境启动脚本

这套方案足以支撑 NeverMind 在早期阶段实现稳定、高效、低阻塞的前后端协作。
